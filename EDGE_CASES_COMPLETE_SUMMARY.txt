Embedding Models:
─────────────────────────────────────────────────────────────────────

✅ What is an embedding?
   → Learned representation in continuous vector space
   → Each dimension captures semantic features
   → Similar texts → Similar vectors

✅ How are they created?
   Step 1: Tokenization (text → tokens)
   Step 2: Token embeddings (tokens → vectors)
   Step 3: Transformer (context processing)
   Step 4: Pooling (single vector output)

✅ Dense vs Sparse
   Dense:  768d, 100% non-zero, 3GB storage
   Sparse: 30K dimensions, 1% non-zero, 0.3GB storage
   Winner: Depends on use case!

✅ When they fail
   • Out-of-vocabulary words
   • Domain-specific jargon
   • Negation handling
   • Numerical values
   • Code snippets

Vector Database:
─────────────────────────────────────────────────────────────────────

✅ What is it?
   → Specialized DB for high-dimensional vectors
   → Fast similarity search (k-NN)
   → Combines with traditional filters

✅ OpenSearch structure
   • Inverted index (text fields) + Vector index (HNSW)
   • Both queryable together (hybrid search!)

✅ Storage per document (1M docs, 768d)
   • Text: ~1-10KB
   • Vector: 3KB (768 × 4 bytes)
   • HNSW edges: ~128 bytes
   • Total: ~4-13KB per doc

HNSW Algorithm:
─────────────────────────────────────────────────────────────────────

✅ What is it?
   → Hierarchical Navigable Small Worlds
   → Multi-layer graph structure
   → Like highway system (fast long jumps, precise local navigation)

✅ How it works
   Layer 2 (top):    Few nodes, long connections
   Layer 1 (middle): More nodes, medium connections  
   Layer 0 (bottom): All nodes, local connections
   
   Search: Start top → Navigate down → Find nearest

✅ Parameters
   M:              Graph connectivity (8-32)
   ef_construction: Build quality (128-512)
   ef_search:      Query accuracy (50-200) ⭐ KEY!

✅ Performance
   Brute force: O(N) → 500ms-5s for 1M vectors
   HNSW:        O(log N) → 10-50ms for 1M vectors
   Speed-up:    10-100x faster!

Distance Metrics:
─────────────────────────────────────────────────────────────────────

✅ Cosine Similarity (Most common for text)
   • Measures angle, not distance
   • Range: [-1, 1]
   • Magnitude invariant
   • [1,1] vs [2,2] = 1.0 (identical direction)

✅ Euclidean Distance (L2)
   • Measures straight-line distance
   • Range: [0, ∞)
   • Magnitude matters
   • [1,1] vs [2,2] = 1.41 (different!)

✅ Dot Product
   • Combines angle + magnitude
   • Range: [-∞, ∞]
   • If normalized, equals cosine

Search Methods:
─────────────────────────────────────────────────────────────────────

✅ Keyword (BM25)
   Speed: 1-5ms
   Quality: Exact match only
   Fails: Synonyms, conceptual queries

✅ Semantic (k-NN)
   Speed: 10-50ms
   Quality: Understands meaning
   Fails: Exact codes, numbers

✅ Hybrid (Combines both) ⭐ BEST
   Speed: 15-60ms
   Quality: State-of-the-art
   Wins: Most use cases

✅ Agentic (Future)
   Speed: 100ms+
   Quality: Natural language
   Uses: AI agent planning

Optimization:
─────────────────────────────────────────────────────────────────────

✅ The Triangle (pick 2):
   COST ↔ RECALL ↔ LATENCY

✅ Cost reduction (97% possible!):
   FP16:    50% savings, 0.5% recall loss
   INT8:    75% savings, 1% recall loss
   Binary:  97% savings, <2% recall loss

✅ Latency optimization:
   Faster model:     25ms → 15ms saved
   GPU:              40ms → 5ms saved (87% faster)
   Cache:            40ms saved (cached queries)
   Lower ef_search:  7ms saved
   Force merge:      5ms saved
   Combined:         60ms → 18ms (70% faster!)

✅ Recall improvement:
   Higher ef_search: +2-5% recall
   Hybrid search:    +10-15% recall
   Fine-tuning:      +10-15% recall
   Reranking:        +5-10% recall
